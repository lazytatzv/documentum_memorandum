# RESOURCES

## 情報収集するということ

素のプログラミング能力や小手先のコマンド芸よりも情報収集能力の方が大切なのではないかと思っている.
有名どころのサイトやリソース元を知っておくことは非常に役立つだろう.

## Links (汎用)

1. [StackOverflow](https://stackoverflow.com/questions)

世界中であまりにも有名すぎるエンジニア向け質問サイト.

2. [GitHub](https://github.com/)

自分でコードを上げる以外にも、他人のIssueを参考にしたり、repoを覗いたりと付き合いは多い.

3. [Qiita](https://qiita.com/)

日本語圏で一番有名なエンジニアブログサイト. 皆使ってる.

4. [Zenn](https://zenn.dev/)

こちらも日本語. Qiitaの次に有名くらい.

5. [Reddit](https://www.reddit.com/)

アメリカで有名なSNS. サイト内検索は微妙なので検索エンジンから情報を探すべき.
エンジニア特化でもなければ情報の信頼性も微妙だったりするので注意.

6. [X/Twitter](https://x.com/home)

情報を探すのにはおすすめしない. トレンド掴むにはおすすめ.

7. [dev.to](https://dev.to/)

英語サイト. 微妙な記事も多いが個人的によくみる. Qiitaっぽい.

## AIを使うことについて

勿論、最近のAIは情報を勝手にSearchしてくれるし、精度も高いため積極的に頼って良いとは思うが、
自分で検索して解決しなければならないことも多々あるため、自分で情報を集めることに慣れることは大切だと考えている.

## 実際のところコード書くときによく見るリソース

有名どころのサイトは挙げたが、実際コーディング中に見るのは、
専らライブラリやフレームワークののドキュメントやセンサーのデータシートなどである.

また、Linksに載っているのはあくまで汎用サイトであり、それぞれの言語やフレームワークのforumを見にいくことも多い.
例えばrosだったらros discourceなど.

所謂関数の使い方が載っている公式のAPI referenceやUsageが載っているDocumentationを読めることは大切.
たまにまともなDocsがない場合があるが、そういう場合はGitHubのREADME.mdやMarkDownでかかれた情報を見たり、
examplesがあることが多いので流用したり、最悪ソースコードを読む、といった感じ. 間違ってもいきなりソースコードを読んだりしない.

逆に、自分がプロジェクトを作るときにDocs整備はかなり重要なので意識する.
あとDocsは.mdで残すこと. .txtとか使わない. 大規模なら.rst(restructuredText)でもありっちゃあり.
.rstはPythonやLinuxコミュニティでよく使われ、Sphinxというツールを使うと綺麗に纏めることができる.

## サイト内検索やCtrl + fを積極的に使う

サイトの中で特定のキーワードを探すことは割と重要. 一般にCtrl+fを使うことでサイト内検索をすることができる.
GitHubなどはサイトの中で検索が強いのでサイトの機能を積極的に活用していく.

そこまでページ数が多くない場合はwgetで落としてきてgrepなどで情報を抜いたり、AIに投げてみるのもアリ.

## 英語に怖気づかない

英語の情報の方が圧倒的に多いため、日本語で見つからない情報も積極的に英語で調べる癖をつける.
専門用語は英語でも言えるようにしておく. そこに英語があるから読むのではなく、自分で英語を使って調べていく.
わからないなら翻訳を使えばいい.

## 一次情報を優先に

一次情報をなるべく参照する. 一生Qiitaだけ見てても限界がある.
また、目的を必ず持って読むこと. だらだら読んでも意味ない.

ただし、自身のレベルや時間制約、諸々考えて臨機応変に参照していくこと.

1. 公式Docs

2. 仕様書, DataSheet

3. GitHub Issue/PR

4. 技術ブログ

5. そのへんの記事、reddit
